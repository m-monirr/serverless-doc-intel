Phase 1  —  Foundation
Days 1–4

Goal
By end of day 4 you can upload a PDF with curl, it gets OCR'd by Docling and split into chunks, a job_id comes back immediately, and you can poll /status to see done_chunks. Redis tracks every job. No Beam, no LLM — just clean plumbing that works end-to-end.

Day 1 — project scaffold
Create the repo: async-pdf-pipeline
Set up virtual environment and install requirements.txt
Copy .env.example to .env — fill in REDIS_URL (local: redis://localhost:6379)
Start local Redis: docker run -d -p 6379:6379 redis:7-alpine
Create the folder structure: api/, beam/, frontend/, tests/
Write FastAPI skeleton in main.py — POST /ingest returns hardcoded job_id for now
Confirm: uvicorn api.main:app --reload starts without errors

Day 2 — tracker.py (all Redis helpers)
Write the complete tracker.py — all three groups of helpers in one file from day one
Job group: init_job(), push_result(), get_job(), set_final_output()
Rate limit group: check_rate_limit(), record_upload(), get_rate_limit_status()
MD5 group: calculate_file_md5(), get_cached_result(), cache_result_by_md5()
Test each function manually in a Python shell — verify keys appear in Redis
Confirm: RPUSH + HINCRBY pipeline is atomic, no race conditions

Day 3 — chunker.py (Docling + splitter)
Write chunker.py — DocumentConverter at module level (NOT inside the function)
Run the file once standalone to trigger Docling model download (~1.5GB, one-time)
Implement extract_and_chunk() — Docling convert → export_to_markdown() → RecursiveCharacterTextSplitter
Test on a real PDF: python -c "from api.chunker import extract_and_chunk; print(extract_and_chunk('test.pdf'))"
Verify chunks are clean, section headers preserved, chunk_id starts at 0

Day 4 — wire everything + end-to-end test
Complete main.py: read file with await file.read(), check magic bytes (%PDF), MD5 check, extract_and_chunk(), init_job()
Add sequential for-loop placeholder (process_chunk_local stub that just writes dummy result to Redis)
Add _watch_and_aggregate stub — just polls until done, logs 'ready to aggregate'
Add GET /status, GET /result, GET /quota endpoints
Test full flow with curl — upload PDF, get job_id, poll status, see done_chunks count up
Test rate limit: upload 3 times, third should return 429
Test MD5: upload same file twice, second should return cached result instantly

Deliverable: curl POST /ingest returns job_id. GET /status shows progress. Rate limit and MD5 cache both work. No LLM output yet — placeholder results only.


Phase 2  —  Beam Integration
Days 5–9

Goal
By end of day 9, chunks are dispatched to real Beam containers running in parallel. You can watch done_chunks increment in Redis as each worker finishes. The GPU endpoint (vllm_server.py) is live and returning text. The CPU workers (worker.py) are deployed and calling it. No aggregation yet.

Day 5 — Beam account + CLI setup
Sign up at beam.cloud
pip install beam-client
beam login — follow the browser prompt
Add Beam credentials to .env: BEAM_CLIENT_ID, BEAM_CLIENT_SECRET
Run: beam token create — save output as BEAM_AUTH_TOKEN in .env
Add all secrets to Beam: beam secret create REDIS_URL ..., beam secret create BEAM_AUTH_TOKEN ...
Add placeholder: beam secret create VLLM_ENDPOINT_URL placeholder

Day 6 — vllm_server.py
Write beam/vllm_server.py — @endpoint with T4 GPU, keep_warm_seconds=60
Pin CUDA correctly in Image(): torch==2.3.0+cu118 with --index-url pytorch whl/cu118
Write the generate() function — global _llm loaded once per container
Write call_vllm() HTTP helper and clean_json() fence stripper
Do NOT run locally — vLLM requires a real GPU

Day 7 — deploy GPU endpoint + first cold start
Run: beam deploy beam/vllm_server.py:generate --name qwen-vllm
Beam prints the endpoint URL — copy it
Update the secret: beam secret create VLLM_ENDPOINT_URL https://your-url.app.beam.cloud
Add VLLM_ENDPOINT_URL to local .env
Warm up the endpoint — send a test curl request and WAIT (first call = 60-90s — model downloading)
Confirm: curl returns {"text": "..."} — endpoint is working

Day 8 — worker.py
Write beam/worker.py — @task_queue, CPU only (no GPU), memory=512
Import call_vllm and clean_json from beam.vllm_server
Build the Qwen chat-format prompt inside process_chunk()
Parse JSON response with Pydantic ChunkAnalysis model
Write atomic Redis result with RPUSH + HINCRBY pipeline
Write fallback handler — never stall the job on one bad chunk
Write process_chunk_local() — identical logic, no Beam dispatch, for benchmark mode

Day 9 — deploy worker + swap main.py loop
Run: beam deploy beam/worker.py:process_chunk --name pdf-chunk-processor
In main.py: replace the sequential for-loop with process_chunk.put(job_id=job_id, chunk=chunk) per chunk
Upload a PDF and watch Beam dashboard — N tasks should appear simultaneously
Poll /status and verify done_chunks increments as workers finish
Compare timing: note how fast parallel is vs the sequential stub from Phase 1

Deliverable: Upload PDF → N Beam tasks fire simultaneously → Redis done_chunks counts up in real time. Each chunk has a real LLM summary from Qwen2.5-7B. GPU cost per run: ~$0.002.


Phase 3  —  LLM Output
Days 10–13

Goal
By end of day 13, GET /result returns a complete structured JSON — abstract, top 10 key points, and 4-section documentation (intro, methods, findings, conclusion). MD5 cache is verified working. Rate limit is tested. The project is functionally complete.

Day 10 — aggregator.py
Write beam/aggregator.py — import get_job, set_final_output, cache_result_by_md5, call_vllm, clean_json
Define FinalOutput Pydantic model at module level (not inside function)
Sort results by int(chunk_id) — explicit cast for JSON round-trip safety
Build digest string from sorted chunk summaries
Write the final Qwen prompt — request abstract + top_key_points[10] + documentation{intro, methods, findings, conclusion}
Parse response through FinalOutput Pydantic model
Write fallback: if parsing fails, use chunk summaries as key_points
Call set_final_output() then cache_result_by_md5() at the end

Day 11 — _watch_and_aggregate in main.py
Replace the stub watcher with the real _watch_and_aggregate() async function
Use asyncio.sleep(2) in the polling loop — NOT time.sleep (blocks the event loop)
Call aggregate() via await loop.run_in_executor(_executor, aggregate, job_id) — non-blocking
Add 300s max_wait timeout — if exceeded, set status='error' in Redis
Create ThreadPoolExecutor at module level: _executor = ThreadPoolExecutor(max_workers=4)
Do the same for _run_sequential — also use run_in_executor for blocking LLM calls

Day 12 — full pipeline test on real papers
Download 3 real research papers (arxiv has free PDFs)
Upload each one and wait for full completion
Inspect the JSON output — check abstract quality, key points relevance, documentation sections
Verify done_chunks == total_chunks before result is available
Verify failed_chunks count is reported correctly if any chunk fails
Check Redis keys after each run — confirm TTLs are set correctly

Day 13 — MD5 + rate limit verification
Re-upload the same paper from day 12 — confirm instant return with cached:true
Verify the re-upload does NOT increment the rate limit counter
Upload 2 different papers — confirm both succeed
Upload a 3rd different paper — confirm 429 with resets_in_minutes in the response
Call GET /quota — verify uploads_used and resets_in_minutes are accurate
Wait for TTL expiry (or manually delete Redis key) and confirm limit resets

Deliverable: GET /result returns full structured JSON. Same PDF uploaded twice returns instantly. Third different PDF returns 429. Project is functionally complete — everything works.


Phase 4  —  Polish & Deploy
Days 14–21

Goal
By end of day 21 the project has a live public URL, a Streamlit frontend anyone can use, a README with real benchmark numbers from your machine, and a demo GIF. This is what goes on your portfolio and GitHub profile.

Day 14 — SSE /stream endpoint
Add GET /stream/{job_id} to main.py using StreamingResponse
Generator function: poll Redis every 0.5s, yield new chunk results as SSE events
Track seen chunk_ids in a set — only yield each chunk once
Break on status=='done' — yield final output event
Break on status=='error' — yield error event
Add 300s timeout — break and stop the generator
Test with curl --no-buffer http://localhost:8000/stream/{job_id}

Day 15 — Streamlit frontend
Write frontend/app.py
File uploader widget — accepts PDF only
On upload: POST to /ingest, store job_id in session state
Progress bar: poll /status every second, update bar with done_chunks/total_chunks
Results panel: show abstract, top key points as a list, documentation as expandable sections
Quota display: show GET /quota result — uploads remaining + reset time
Run: streamlit run frontend/app.py

Day 16 — benchmark.py
Run benchmark.py against a real 40-page research paper
Record both times: parallel and sequential
Calculate speedup: sequential_time / parallel_time
This number goes in your README title — write it down
Run 3 times and take the average — first run may be slower due to GPU warm-up

Day 17 — production deployment
Upstash Redis: go to upstash.com, create database, copy production REDIS_URL
Update Beam secret: beam secret create REDIS_URL <production-upstash-url>
Deploy FastAPI to Railway: push repo to GitHub, connect in Railway, set env vars
Set all env vars in Railway: REDIS_URL, VLLM_ENDPOINT_URL, BEAM_CLIENT_ID, BEAM_CLIENT_SECRET, BEAM_AUTH_TOKEN
Test the live URL end-to-end — upload a PDF through the production endpoint

Days 18–19 — README + demo assets
README headline: 'Processed a 40-page PDF in Xs (parallel) vs Ys (sequential) — Zx speedup'
Architecture section: draw the system diagram (use Excalidraw or screenshot the doc diagram)
Tech stack table: Docling, vLLM, Beam, Qwen2.5-7B, Redis, FastAPI
Setup instructions: pip install, beam deploy order, env vars
Record demo GIF: use Gifox or ScreenToGif — show upload → progress bar → results appearing
Deployment section: link to live URL

Days 20–21 — buffer + final cleanup
Handle edge cases: empty PDF, corrupted file, PDF with only images and no OCR
Add file size validation message to Streamlit UI
Clean up any debug print statements
Make sure all secrets are in .env.example (without values)
Final GitHub push — clean commit history, good commit messages
Share the repo URL

Deliverable: Live URL. Streamlit demo. README with real benchmark numbers and architecture diagram. Demo GIF. Clean GitHub repo. Portfolio-ready.

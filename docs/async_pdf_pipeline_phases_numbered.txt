  27: Phase 1  —  Foundation
  28: Days 1–4
  29: 
  30: Goal
  31: By end of day 4 you can upload a PDF with curl, it gets OCR'd by Docling and split into chunks, a job_id comes back immediately, and you can poll /status to see done_chunks. Redis tracks every job. No Beam, no LLM — just clean plumbing that works end-to-end.
  32: 
  33: Day 1 — project scaffold
  34: Create the repo: async-pdf-pipeline
  35: Set up virtual environment and install requirements.txt
  36: Copy .env.example to .env — fill in REDIS_URL (local: redis://localhost:6379)
  37: Start local Redis: docker run -d -p 6379:6379 redis:7-alpine
  38: Create the folder structure: api/, beam/, frontend/, tests/
  39: Write FastAPI skeleton in main.py — POST /ingest returns hardcoded job_id for now
  40: Confirm: uvicorn api.main:app --reload starts without errors
  41: 
  42: Day 2 — tracker.py (all Redis helpers)
  43: Write the complete tracker.py — all three groups of helpers in one file from day one
  44: Job group: init_job(), push_result(), get_job(), set_final_output()
  45: Rate limit group: check_rate_limit(), record_upload(), get_rate_limit_status()
  46: MD5 group: calculate_file_md5(), get_cached_result(), cache_result_by_md5()
  47: Test each function manually in a Python shell — verify keys appear in Redis
  48: Confirm: RPUSH + HINCRBY pipeline is atomic, no race conditions
  49: 
  50: Day 3 — chunker.py (Docling + splitter)
  51: Write chunker.py — DocumentConverter at module level (NOT inside the function)
  52: Run the file once standalone to trigger Docling model download (~1.5GB, one-time)
  53: Implement extract_and_chunk() — Docling convert → export_to_markdown() → RecursiveCharacterTextSplitter
  54: Test on a real PDF: python -c "from api.chunker import extract_and_chunk; print(extract_and_chunk('test.pdf'))"
  55: Verify chunks are clean, section headers preserved, chunk_id starts at 0
  56: 
  57: Day 4 — wire everything + end-to-end test
  58: Complete main.py: read file with await file.read(), check magic bytes (%PDF), MD5 check, extract_and_chunk(), init_job()
  59: Add sequential for-loop placeholder (process_chunk_local stub that just writes dummy result to Redis)
  60: Add _watch_and_aggregate stub — just polls until done, logs 'ready to aggregate'
  61: Add GET /status, GET /result, GET /quota endpoints
  62: Test full flow with curl — upload PDF, get job_id, poll status, see done_chunks count up
  63: Test rate limit: upload 3 times, third should return 429
  64: Test MD5: upload same file twice, second should return cached result instantly
  65: 
  66: Deliverable: curl POST /ingest returns job_id. GET /status shows progress. Rate limit and MD5 cache both work. No LLM output yet — placeholder results only.

  69: Phase 2  —  Beam Integration
  70: Days 5–9
  71: 
  72: Goal
  73: By end of day 9, chunks are dispatched to real Beam containers running in parallel. You can watch done_chunks increment in Redis as each worker finishes. The GPU endpoint (vllm_server.py) is live and returning text. The CPU workers (worker.py) are deployed and calling it. No aggregation yet.
  74: 
  75: Day 5 — Beam account + CLI setup
  76: Sign up at beam.cloud
  77: pip install beam-client
  78: beam login — follow the browser prompt
  79: Add Beam credentials to .env: BEAM_CLIENT_ID, BEAM_CLIENT_SECRET
  80: Run: beam token create — save output as BEAM_AUTH_TOKEN in .env
  81: Add all secrets to Beam: beam secret create REDIS_URL ..., beam secret create BEAM_AUTH_TOKEN ...
  82: Add placeholder: beam secret create VLLM_ENDPOINT_URL placeholder
  83: 
  84: Day 6 — vllm_server.py
  85: Write beam/vllm_server.py — @endpoint with T4 GPU, keep_warm_seconds=60
  86: Pin CUDA correctly in Image(): torch==2.3.0+cu118 with --index-url pytorch whl/cu118
  87: Write the generate() function — global _llm loaded once per container
  88: Write call_vllm() HTTP helper and clean_json() fence stripper
  89: Do NOT run locally — vLLM requires a real GPU
  90: 
  91: Day 7 — deploy GPU endpoint + first cold start
  92: Run: beam deploy beam/vllm_server.py:generate --name qwen-vllm
  93: Beam prints the endpoint URL — copy it
  94: Update the secret: beam secret create VLLM_ENDPOINT_URL https://your-url.app.beam.cloud
  95: Add VLLM_ENDPOINT_URL to local .env
  96: Warm up the endpoint — send a test curl request and WAIT (first call = 60-90s — model downloading)
  97: Confirm: curl returns {"text": "..."} — endpoint is working
  98: 
  99: Day 8 — worker.py
 100: Write beam/worker.py — @task_queue, CPU only (no GPU), memory=512
 101: Import call_vllm and clean_json from beam.vllm_server
 102: Build the Qwen chat-format prompt inside process_chunk()
 103: Parse JSON response with Pydantic ChunkAnalysis model
 104: Write atomic Redis result with RPUSH + HINCRBY pipeline
 105: Write fallback handler — never stall the job on one bad chunk
 106: Write process_chunk_local() — identical logic, no Beam dispatch, for benchmark mode
 107: 
 108: Day 9 — deploy worker + swap main.py loop
 109: Run: beam deploy beam/worker.py:process_chunk --name pdf-chunk-processor
 110: In main.py: replace the sequential for-loop with process_chunk.put(job_id=job_id, chunk=chunk) per chunk
 111: Upload a PDF and watch Beam dashboard — N tasks should appear simultaneously
 112: Poll /status and verify done_chunks increments as workers finish
 113: Compare timing: note how fast parallel is vs the sequential stub from Phase 1
 114: 
 115: Deliverable: Upload PDF → N Beam tasks fire simultaneously → Redis done_chunks counts up in real time. Each chunk has a real LLM summary from Qwen2.5-7B. GPU cost per run: ~$0.002.

 118: Phase 3  —  LLM Output
 119: Days 10–13
 120: 
 121: Goal
 122: By end of day 13, GET /result returns a complete structured JSON — abstract, top 10 key points, and 4-section documentation (intro, methods, findings, conclusion). MD5 cache is verified working. Rate limit is tested. The project is functionally complete.
 123: 
 124: Day 10 — aggregator.py
 125: Write beam/aggregator.py — import get_job, set_final_output, cache_result_by_md5, call_vllm, clean_json
 126: Define FinalOutput Pydantic model at module level (not inside function)
 127: Sort results by int(chunk_id) — explicit cast for JSON round-trip safety
 128: Build digest string from sorted chunk summaries
 129: Write the final Qwen prompt — request abstract + top_key_points[10] + documentation{intro, methods, findings, conclusion}
 130: Parse response through FinalOutput Pydantic model
 131: Write fallback: if parsing fails, use chunk summaries as key_points
 132: Call set_final_output() then cache_result_by_md5() at the end
 133: 
 134: Day 11 — _watch_and_aggregate in main.py
 135: Replace the stub watcher with the real _watch_and_aggregate() async function
 136: Use asyncio.sleep(2) in the polling loop — NOT time.sleep (blocks the event loop)
 137: Call aggregate() via await loop.run_in_executor(_executor, aggregate, job_id) — non-blocking
 138: Add 300s max_wait timeout — if exceeded, set status='error' in Redis
 139: Create ThreadPoolExecutor at module level: _executor = ThreadPoolExecutor(max_workers=4)
 140: Do the same for _run_sequential — also use run_in_executor for blocking LLM calls
 141: 
 142: Day 12 — full pipeline test on real papers
 143: Download 3 real research papers (arxiv has free PDFs)
 144: Upload each one and wait for full completion
 145: Inspect the JSON output — check abstract quality, key points relevance, documentation sections
 146: Verify done_chunks == total_chunks before result is available
 147: Verify failed_chunks count is reported correctly if any chunk fails
 148: Check Redis keys after each run — confirm TTLs are set correctly
 149: 
 150: Day 13 — MD5 + rate limit verification
 151: Re-upload the same paper from day 12 — confirm instant return with cached:true
 152: Verify the re-upload does NOT increment the rate limit counter
 153: Upload 2 different papers — confirm both succeed
 154: Upload a 3rd different paper — confirm 429 with resets_in_minutes in the response
 155: Call GET /quota — verify uploads_used and resets_in_minutes are accurate
 156: Wait for TTL expiry (or manually delete Redis key) and confirm limit resets
 157: 
 158: Deliverable: GET /result returns full structured JSON. Same PDF uploaded twice returns instantly. Third different PDF returns 429. Project is functionally complete — everything works.
